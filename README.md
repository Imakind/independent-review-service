# Independent Review Service

## Суть проекта

Сервис для независимых отзывов о продавцах, аккаунтах, сервисах и контактах, даже если исходная площадка не поддерживает честную публичную репутацию или удаляет негативные отзывы.

Примеры объектов отзыва:

- аккаунт Instagram;
- профиль OLX или другой маркетплейс;
- Telegram-аккаунт;
- номер телефона;
- сайт, магазин, сервис или компания.

Главная идея: пользователь может проверить внешний идентификатор и увидеть историю отзывов вне исходной платформы.

## Проблема

- На отдельных площадках отзывы могут модерироваться непрозрачно или удаляться.
- Мошенники могут менять площадку, но сохранять телефон, никнейм, ссылку или другой идентификатор.
- У пользователя часто нет единого места для проверки репутации продавца или контакта.

## Цель

Создать независимый слой репутации:

1. Пользователь вводит идентификатор: номер, ссылку, username, домен или название.
2. Сервис нормализует идентификатор и ищет совпадения.
3. Пользователь видит агрегированную карточку: рейтинг, отзывы, жалобы, подтверждения, даты.
4. Пользователь может оставить собственный отзыв с доказательствами.

## MVP

Первая версия реализуется как Telegram-бот.

### Основные сценарии

- Проверить объект:
  - по номеру телефона;
  - по ссылке на профиль;
  - по username;
  - по названию сервиса.
- Оставить отзыв:
  - тип: положительный, нейтральный, негативный;
  - текст;
  - категория проблемы;
  - опциональные доказательства: скриншот, ссылка, описание сделки.
- Посмотреть карточку объекта:
  - общий рейтинг;
  - количество отзывов;
  - последние отзывы;
  - предупреждения о рисках.
- Пожаловаться на отзыв:
  - спам;
  - клевета;
  - персональные данные;
  - дубликат;
  - недостаточно доказательств.

## Telegram-бот: начальная реализация

### Команды

- `/start` — запуск и краткое описание сервиса.
- `/check` — проверка идентификатора.
- `/review` — создание отзыва.
- `/help` — справка.
- `/rules` — правила публикации.

### Поток проверки

1. Бот запрашивает идентификатор.
2. Система определяет тип идентификатора.
3. Идентификатор приводится к нормальной форме.
4. Выполняется поиск по базе.
5. Бот возвращает карточку объекта или сообщает, что данных пока нет.

### Поток создания отзыва

1. Пользователь отправляет идентификатор объекта.
2. Выбирает тип отзыва.
3. Указывает категорию.
4. Пишет текст.
5. Прикрепляет доказательства при наличии.
6. Подтверждает публикацию.
7. Отзыв сохраняется со статусом `pending` или `published` в зависимости от правил модерации.

## Архитектура

### Компоненты

- Telegram Bot API — пользовательский интерфейс MVP.
- Backend API — бизнес-логика, нормализация, поиск, модерация.
- Database — хранение объектов, отзывов, пользователей и жалоб.
- Moderation module — проверка жалоб, спама, дублей и опасного контента.
- Admin panel — ручная модерация и управление спорными отзывами.

### Предлагаемый стек

- Backend: Node.js / TypeScript или Python.
- Bot framework: Telegraf, grammY или aiogram.
- Database: PostgreSQL.
- Search: PostgreSQL full-text search на MVP, позже Elasticsearch / OpenSearch.
- Storage: S3-совместимое хранилище для доказательств.
- Deployment: Docker Compose на MVP, позже Kubernetes или managed platform.

## Модель данных

### Object

Объект, о котором оставляют отзывы.

- `id`
- `type`: `phone`, `social_profile`, `marketplace_profile`, `website`, `service`, `person_alias`
- `normalized_identifier`
- `display_identifier`
- `created_at`
- `updated_at`

### Review

Отзыв пользователя.

- `id`
- `object_id`
- `author_user_id`
- `rating`: `positive`, `neutral`, `negative`
- `category`
- `text`
- `evidence_refs`
- `status`: `pending`, `published`, `rejected`, `hidden`
- `created_at`
- `updated_at`

### Report

Жалоба на отзыв.

- `id`
- `review_id`
- `reporter_user_id`
- `reason`
- `comment`
- `status`
- `created_at`

## Нормализация идентификаторов

Нормализация нужна, чтобы один и тот же объект не создавался много раз.

Примеры:

- телефон приводится к единому международному формату;
- ссылки очищаются от UTM-параметров;
- username приводится к нижнему регистру, если платформа не различает регистр;
- домен очищается от протокола и лишних слешей.

## Модерация и достоверность

Сервис не должен превращаться в площадку для клеветы, доксинга или спама.

Обязательные правила:

- запрещать публикацию лишних персональных данных;
- отделять факт сделки от субъективной оценки;
- давать возможность оспорить отзыв;
- помечать отзывы без доказательств;
- ограничивать массовые публикации с новых аккаунтов;
- хранить историю модерации.

## Правовые и этические ограничения

Проект работает с потенциально чувствительными данными: номерами телефонов, профилями, обвинениями в мошенничестве и пользовательскими доказательствами.

До публичного запуска требуется отдельная юридическая проверка:

- правил обработки персональных данных;
- правил хранения доказательств;
- политики удаления данных;
- процедуры оспаривания отзывов;
- ответственности за пользовательский контент.

## Приоритеты разработки

1. Создать Telegram-бота с базовыми командами.
2. Реализовать нормализацию идентификаторов.
3. Добавить базу данных объектов и отзывов.
4. Сделать поиск по идентификатору.
5. Добавить модерацию и жалобы.
6. Добавить админ-панель.
7. Подготовить публичный API и веб-интерфейс.

## Критерии готовности MVP

- Пользователь может проверить номер, ссылку или username.
- Пользователь может оставить отзыв.
- Отзыв сохраняется в базе.
- Поиск возвращает карточку объекта.
- Есть минимальная защита от спама.
- Есть механизм жалобы на отзыв.
- Администратор может скрыть или отклонить спорный отзыв.
